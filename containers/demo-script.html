03

What Are Containers?

8 – 15 min
1
The Problem — "It Works on My Machine"
⌃

Talking point: "Who here has heard 'it works on my machine'? Raise your hand. This is the core problem containers solve. Different OS versions, different library versions, different runtime environments — containers package everything together so the app runs identically everywhere."

💡
Pause for audience show of hands. It warms up the room and makes the problem concrete.
2
VMs vs Containers
⌃

Talking point: "A VM virtualizes the entire hardware stack — you get a full OS guest on top of a hypervisor. A container shares the host kernel and just packages your app and its dependencies. That's why containers start in seconds and are measured in megabytes, while VMs take minutes and gigabytes."

VIRTUAL MACHINE
App + Guest OS + Hypervisor
GBs — minutes to start
vs
CONTAINER
App + Dependencies (shared kernel)
MBs — seconds to start
3
The Shipping Container Analogy
⌃

Talking point: "Think about shipping containers — before standardized steel containers, each cargo ship needed custom handling for every type of cargo. After standardization, any container fits any ship, any truck, any crane. Docker containers did the same thing for software: one standard format that runs on any machine, any cloud, any CI system."

4
Image vs Container
⌃

Talking point: "An image is the blueprint — a read-only snapshot of your app and everything it needs. A container is a running instance of that image. One image can run as many containers as you need. Think of an image like a class definition in code, and a container like an object instance."

5
Why Containers on AWS?
⌃

Talking point: "AWS gives you a managed stack for every layer of the container journey: ECR for storing images, ECS and Fargate for running them without managing servers, and EKS if you need the full Kubernetes control plane. You can go from Dockerfile to production without touching a single EC2 instance."

04

Docker 101

15 – 22 min
1
Walk Through the Dockerfile
⌃

Show the Dockerfile slide, then switch to the terminal and run cat Dockerfile so the audience sees the real file. Three lines is the point — it's intentionally simple.

dockerfile 
FROM nginx:alpine
COPY app/index.html /usr/share/nginx/html/index.html
EXPOSE 80

Talking point line by line:

  • FROM nginx:alpine — start from the official nginx image on Alpine Linux. Tiny base, pre-configured web server included.
  • COPY app/index.html ... — replace nginx's default page with our app. That's our entire "deployment."
  • EXPOSE 80 — documentation that the container listens on port 80. Doesn't actually open a port — that's done at runtime.
Terminal — ~/containers
$ cat Dockerfile
FROM nginx:alpine
COPY app/index.html /usr/share/nginx/html/index.html
EXPOSE 80
2
Core Docker Commands Explained
⌃

Briefly explain the four commands before running them live in Section 6. Set context so the audience knows what to look for.

  • docker build — reads the Dockerfile, downloads layers, produces an image
  • docker images — lists images on your local machine
  • docker run — creates and starts a container from an image
  • docker push — uploads the image to a registry (ECR)
✅
Remind the audience: we'll run all of these live in a few minutes. This is just the mental model.
05

AWS Services Overview

22 – 35 min (slides)
1
ECR — Elastic Container Registry
⌃

Talking point: "ECR is AWS's fully managed Docker registry. Think of it as a private Docker Hub, but inside your AWS account. Images are encrypted at rest, integrated with IAM, and automatically scanned for CVEs on push. No quota limits, no rate limiting on pulls from your own account."

2
ECS — Elastic Container Service
⌃

Talking point: "ECS is AWS's container orchestrator. You define a Task Definition — what image, how much CPU and memory, what ports — then ECS handles scheduling, health checks, and restarts. With the Fargate launch type, AWS manages the underlying compute entirely. You never see an EC2 instance."

3
EKS — Elastic Kubernetes Service
⌃

Talking point: "EKS runs a managed Kubernetes control plane. AWS handles etcd, the API server, upgrades, and HA. You own the worker nodes — or use EKS Auto Mode, which is the 2025 default that makes Kubernetes nearly ops-free. Use EKS when you need the full Kubernetes ecosystem, multi-cloud portability, or already have Kubernetes expertise on your team."

💡
EKS Auto Mode (2025): Makes Kubernetes operations-free by handling node provisioning, scaling, and lifecycle automatically. Mention this as the modern starting point for new EKS workloads.
4
App Runner — Fully Managed Container Hosting
⌃

Talking point: "App Runner is the simplest option: point it at an ECR image or a GitHub repo, and it handles everything — build, deploy, scaling, load balancing, TLS. Zero infrastructure decisions. Perfect for internal tools, APIs, and side projects where you want zero ops overhead."

💡
2025 App Runner updates: AWS shipped major improvements including VPC support (connect to RDS, ElastiCache), WAF integration for security, and custom domain management. App Runner is now viable for production workloads that previously required ECS.

LIVE: Build + Push to ECR

22 – 35 min · Live Demo
A
Step 6a — Build the Image Locally
⌃

Run each command in sequence. Narrate as you type. Pause after docker images so the audience can see the image size.

bash 
# Build the image — run from containers/ directory
docker build -t containers-demo .

# Confirm the image was created
docker images | grep containers-demo

# Run it locally and test
docker run -d -p 8080:80 --name demo-local containers-demo
# Open http://localhost:8080 in browser

# Stop and remove the local container
docker stop demo-local && docker rm demo-local
Terminal — ~/containers
$ docker build -t containers-demo .
[+] Building 4.2s (7/7) FINISHED
$ docker images | grep containers-demo
containers-demo latest a1b2c3d4e5f6 5 seconds ago 42MB
$ docker run -d -p 8080:80 --name demo-local containers-demo
7f3e9c2a1b4d...

Talking point: "About 40MB — tiny compared to a full EC2 AMI which is typically 8–25GB. That's the power of Alpine-based images. Open localhost:8080 — you're looking at our app running inside a container on your laptop."

B
Step 6b — Authenticate + Push to ECR
⌃

Walk through each command block. Explain what each variable resolves to as you set it.

bash — set variables 
# Grab account ID and set region
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGION=us-east-1
echo "Account: $AWS_ACCOUNT_ID"
bash — authenticate to ECR 
aws ecr get-login-password --region $REGION \
  | docker login --username AWS --password-stdin \
    $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com
bash — tag + push 
# Tag the image with the full ECR URI
docker tag containers-demo:latest \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest

# Push to ECR
docker push \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest
Terminal — ECR push output
$ aws ecr get-login-password --region us-east-1 | docker login ...
Login Succeeded
$ docker push ...amazonaws.com/containers-demo:latest
latest: digest: sha256:a1b2c3... size: 942
Pushed successfully
✅
ECR auth tip: The token returned by get-login-password is valid for 12 hours. If you get auth errors, simply re-run the login command — do not try to debug Docker credentials further.

After push: switch to the ECR console tab and show the image in the repository. Click into it to show the digest, size, and scan results.

EKS vs App Runner

52 – 57 min
1
When to Choose EKS
⌃

Talking point: "Choose EKS when you need the Kubernetes ecosystem — Helm charts, custom controllers, service mesh, or multi-cloud portability. EKS Auto Mode (launched in 2025) handles node provisioning and lifecycle automatically, making it the right starting point for new EKS workloads. It's no longer the 'hard' choice."

  • Multi-cloud portability (Kubernetes runs anywhere)
  • Team already has Kubernetes expertise
  • Need custom CRDs, operators, or advanced networking
  • Large-scale microservices (>20 services)
💡
EKS Auto Mode (2025): Eliminates the need to manage node groups, instance types, and AMI upgrades. Mention this as the modern EKS starting point — it removes the main historical objection to EKS (operational complexity).
2
When to Choose App Runner
⌃

Talking point: "Choose App Runner when you want zero infrastructure decisions. Point it at an ECR image, set a port, and you're live — with auto-scaling, load balancing, and TLS included. The 2025 updates added VPC support (so you can connect to RDS and ElastiCache), WAF integration, and custom domain management. App Runner is now viable for real production APIs, not just side projects."

  • Web services and REST APIs
  • Internal tools and dashboards
  • Teams that want no ops overhead
  • Stateless workloads with predictable traffic
3
Decision Framework
⌃

Talking point: "Here's how I think about it: start with App Runner if you want the least friction. Move to ECS Fargate when you need more control — VPC routing, service discovery, load balancer integration. Move to EKS when you need the Kubernetes ecosystem or have 20+ microservices. There's no wrong answer — pick the simplest one that meets your requirements."

09

Recap + Q&A

57 – 60 min
1
5 Key Takeaways
⌃

Read each takeaway aloud, pausing briefly after each one to let it land.

  • Containers are lightweight VMs — same isolation, but MBs not GBs, and seconds not minutes to start.
  • A Dockerfile is all you need — three lines can get you a production-ready container image.
  • ECR is your private registry — integrated with IAM, encrypted, and scanned automatically.
  • ECS Fargate = no servers — define what you want to run, not where to run it. AWS handles the rest.
  • Right service for the right job — App Runner for zero ops, ECS for control, EKS for Kubernetes scale.
2
Q&A — Open Floor
⌃

Open the floor for questions. Common questions and quick answers:

  • "How much does this cost?" — The Fargate task we just ran costs ~$0.024/hour at 0.5 vCPU / 1 GB. ECR storage is $0.10/GB/month. Clean up after the demo and it's effectively free.
  • "Can I run multiple containers?" — Yes. Add multiple container definitions to the Task Definition. Use ECS Service Connect or a load balancer for inter-service communication.
  • "How do I update the app?" — Build a new image with a new tag, push to ECR, update the task definition to point to the new tag, then trigger a new deployment on the service.
  • "What about databases?" — Containers are stateless by design. Use RDS, Aurora, DynamoDB, or ElastiCache for persistent state. Mount EFS volumes if you need shared file storage.
✅
If a question goes deep, offer to follow up after the session. Don't let one question consume the entire Q&A window.

Cleanup

After the Session
1
Delete ECR Images
⌃

Run this command to delete the pushed image from ECR. The repository itself will remain (empty), which is free.

bash 
# Delete ECR image
aws ecr batch-delete-image \
  --repository-name containers-demo \
  --region us-east-1 \
  --image-ids imageTag=latest
⚠️
Always clean up after the demo. A 0.5 vCPU / 1 GB Fargate task costs approximately $0.024/hour while running. Left running overnight, that's ~$0.58. Clean up the ECS service and cluster from the console after the session.
2
Delete ECS Service + Cluster
⌃

ECS services and clusters must be deleted from the console in this order:

  1. In ECS console → your cluster → Services tab → select containers-demo-svc → Update → set Desired tasks to 0 → Save
  2. Wait for running task count to reach 0 (30–60 seconds)
  3. Select the service → Delete → confirm
  4. Click Clusters → select containers-demo-cluster → Delete cluster → confirm
✅
The Task Definition is free to keep — it has no ongoing cost. You can delete it from Task Definitions → select it → Deregister if you want a clean account.
3
Clean Up Local Docker Images
⌃

Optional — remove local images to free up disk space.

bash 
# Remove local containers-demo image
docker rmi containers-demo:latest

# Remove the tagged ECR copy too
docker rmi $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest

# Optional: prune dangling images
docker image prune -f

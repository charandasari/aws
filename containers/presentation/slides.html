💡 Code Analogy

An Image is like a class in code — a definition, a blueprint. A Container is like an object — an instance of that class, running in memory. One image can spawn many containers simultaneously.

✅ Why This Matters

You build the image once. That exact same image runs on your laptop, CI system, and production — guaranteed to behave identically. No more environment drift.

Section 2 · 8–15 min

Docker Images — Layers All the Way Down

A Docker image is built from layers. Each instruction in a Dockerfile adds a layer. Layers are cached — only changed layers are rebuilt. This is what makes Docker builds fast.

📋 Image Layers (Bottom to Top)
  • ▸Base layer — FROM nginx:alpine — the foundation OS/runtime
  • ▸Dependency layer — RUN npm install — installed packages
  • ▸App layer — COPY . . — your source code
  • ▸Config layer — ENV, EXPOSE, CMD — runtime config
✅ Layer Caching Rule

Put things that change least often at the top of your Dockerfile. Put your app code last. This way, npm install is only re-run when package.json changes — not on every code edit.

The Dockerfile — Our App's Blueprint

# Start from the official nginx image
# alpine = lightweight Linux variant (~5MB)
FROM nginx:alpine

# Copy our HTML app into the nginx web root
COPY index.html /usr/share/nginx/html/index.html

# Tell Docker this container listens on port 80
EXPOSE 80

# Command to run when the container starts
CMD ["nginx", "-g", "daemon off;"]
✅ Zero Code Changes

The HTML app in this repo works without modification. We just wrap it in a container. Same index.html — now portable everywhere.

Dockerfile Instructions

  • ▸FROM — base image. Everything builds on top of this. nginx:alpine gives us a web server in ~5MB.
  • ▸COPY — copy files from your local machine into the image. Source → Destination.
  • ▸RUN — execute a shell command during build. Use for npm install, pip install, etc.
  • ▸EXPOSE — document which port the app listens on. Informational — doesn't publish the port automatically.
  • ▸ENV — set environment variables available at runtime.
  • ▸CMD — the default command to run when the container starts. Only one CMD per Dockerfile.

🗄 What Is a Registry?

A registry is a storage service for Docker images. Like GitHub for code — but for container images. Docker Hub is the public default. ECR is AWS's private, managed alternative. Images are tagged by version: my-app:v1, my-app:latest.

✅ Why ECR Instead of Docker Hub?

ECR images stay inside AWS — no internet egress to pull them from ECS. IAM controls who can push or pull. Built-in image vulnerability scanning. No rate limiting. Integrates natively with ECS and EKS.

Section 3 · ECR

ECR — Elastic Container Registry

  • ▸Private Docker registry — fully managed by AWS. No servers to run. Pay only for storage and data transfer.
  • ▸IAM-based access control — push/pull permissions controlled via IAM policies. No separate username/password management.
  • ▸Integrated image scanning — scan every pushed image for known OS vulnerabilities (CVEs). Free basic scan; enhanced scan uses Inspector.
  • ▸Lifecycle policies — automatically delete old image versions. Keep only the last 10 tagged images. Saves storage costs.
  • ▸No egress charges from ECS — pulling images from ECR to ECS in the same region is free. No internet traffic.
  • ▸Cross-region replication — replicate images to other regions automatically for multi-region deployments.

# Authenticate Docker to ECR

Section 3 · ECS

ECS — Elastic Container Service

ECS is AWS's native container orchestrator. It manages where containers run, keeps them healthy, and handles scaling — without you needing to know Kubernetes.

Key Concepts

  • ▸Task Definition — the blueprint. Defines which image to run, how much CPU/RAM, ports, env vars, and logging.
  • ▸Task — a running instance of a task definition. Like a docker run but managed by ECS.
  • ▸Service — keeps N copies of a task running. If one dies, ECS starts a replacement automatically.
  • ▸Cluster — the logical grouping of tasks and services. One cluster per environment (dev, prod) is typical.
🖥 EC2 Launch Type

You provision and manage the EC2 instances. ECS schedules containers onto them. You control the instance type, patching, and scaling. More control, more ops work.

⚡ Fargate Launch Type (What We Use)

No EC2 to manage. AWS provisions the compute for each task. You only define CPU and memory. Pay per second while the task runs. Perfect for teams who want to focus on the app, not the infrastructure.

✅ When to Choose ECS

Most production workloads. AWS-native tooling. No Kubernetes expertise needed. Strong integration with ALB, IAM, CloudWatch, and CodeDeploy.

Section 3 · EKS

EKS — Elastic Kubernetes Service

  • ▸Managed Kubernetes — AWS runs the Kubernetes control plane (API server, etcd). You manage the worker nodes (EC2 or Fargate).
  • ▸Full Kubernetes API — every kubectl command works. Deploy Helm charts, use any Kubernetes operator, any CNCF tool.
  • ▸Multi-cloud portability — the same Kubernetes manifests run on EKS, GKE, or AKS. If you ever leave AWS, your workloads come with you.
  • ▸Cost: $0.10/hour per cluster (~$73/month) for the control plane, plus EC2 worker node costs.
  • ▸Best for: teams with existing Kubernetes expertise, complex multi-service workloads, or multi-cloud requirements.
⚠ EKS Complexity Budget

Kubernetes has a real learning curve. If your team doesn't already know Kubernetes, don't start with EKS. Start with ECS Fargate. Move to EKS when you have a specific reason — not because it sounds more impressive.

💡 ECS vs EKS — The One-Line Answer

If you need Kubernetes specifically — use EKS. If you just need to run containers on AWS reliably — use ECS Fargate. ECS is simpler, cheaper to start, and less to operate.

✅ What App Runner Handles

  • ✓TLS certificate — HTTPS out of the box
  • ✓Auto-scaling — scales to zero when no traffic
  • ✓Health checks — replaces unhealthy instances
  • ✓Deployments — push a new image, App Runner rolls it out
⚡ When to Use App Runner
  • ✓Simple web apps or APIs
  • ✓Teams that want zero infrastructure management
  • ✓Prototypes or MVPs — fastest time to production
  • ✓Workloads with variable or unpredictable traffic
⚠ Tradeoff

Less control than ECS. No VPC-level networking config. No task-level IAM roles (uses App Runner service role). Not suitable for complex multi-container workloads.

Section 5 · 35–52 min · Live Demo
Step 3 of 5 — AWS Console

Create ECS Cluster

  1. Go to ECS Console — search "ECS" in AWS Console → Clusters → Create cluster
  2. Cluster name: containers-demo-cluster
  3. Infrastructure: select AWS Fargate (serverless) — do not select EC2
  4. Leave defaults — CloudWatch Container Insights optional but useful
  5. Click Create — takes about 30 seconds. Cluster shows as Active.
💡 What a Cluster Is

A cluster is a logical boundary — it groups tasks and services together. It doesn't provision any compute. With Fargate, the cluster is just a namespace. AWS provisions the underlying compute per task, not per cluster.

Section 5 · Live Demo
Step 4 of 5 — AWS Console

Create Task Definition

  1. ECS Console → Task Definitions → Create new task definition
  2. Launch type: Fargate. OS: Linux/X86_64
  3. Task size: 0.5 vCPU, 1 GB memory
  4. Container name: web — Image URI: paste your ECR image URI (123456789.dkr.ecr.us-east-1.amazonaws.com/containers-demo:v1)
  5. Port mappings: Container port 80, Protocol TCP
  6. Logging: enable AWS CloudWatch Logs (auto-creates log group)
  7. Create — task definition is registered with revision :1
Section 5 · Live Demo
Step 5 of 5 — AWS Console

Create ECS Service + Access the App

  1. Cluster → Services → Create
  2. Launch type: Fargate
  3. Task definition: select the one you just created
  4. Service name: web-service · Desired tasks: 1
  5. VPC: default VPC, public subnet, Auto-assign public IP: Enabled
  6. Security group: create new — allow inbound TCP port 80 from anywhere
  7. Click Create — service launches. Wait ~60 sec for task status: RUNNING
  8. Click the task → copy the Public IP → open in browser

Key Takeaways

  • 1Containers solve environment consistency. Ship the app and its runtime together. No more "it works on my machine."
  • 2Docker builds images. A Dockerfile defines the layers. docker build produces a portable, versioned image.

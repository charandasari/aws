Service Roles

Service Reference
🐳
Docker
⌃

Role: Builds container images from Dockerfiles — packages your application and its runtime into a portable, self-contained unit.

  • Key commands: docker build, docker run, docker push
  • In this demo: packages nginx:alpine + index.html into a ~40MB image

Our Dockerfile for this demo:

dockerfile 
FROM nginx:alpine
COPY app/index.html /usr/share/nginx/html/index.html
EXPOSE 80
📦
Amazon ECR
⌃

Role: Private Docker image registry — stores and versions your container images securely within your AWS account.

  • Pricing: $0.10/GB/month for Standard storage. Free tier: 500MB/month (new accounts, 12 months)
  • Integrates natively with ECS, EKS, CodeBuild — no extra auth configuration needed
  • ECR pull-through cache: cache public images from Docker Hub or ECR Public in your private registry
  • Auth: tokens valid 12 hours — use aws ecr get-login-password to authenticate Docker
ECR Authentication
$ aws ecr get-login-password --region us-east-1 | docker login --username AWS --password-stdin <account>.dkr.ecr.us-east-1.amazonaws.com
Login Succeeded
⚙️
Amazon ECS
⌃

Role: Container orchestration — schedules, runs, and manages containers at scale. ECS decides where and how containers run based on your task definitions and service configuration.

  • Two launch types: EC2 (you manage servers) vs Fargate (AWS manages servers)
  • Core concepts: Cluster → Service → Task Definition → Task
  • Auto-scaling: via CloudWatch alarms on CPU%, memory%, or custom metrics

☁️
AWS Fargate
⌃

Role: Serverless compute for containers — no EC2 instances to provision, patch, or scale. You define what to run; Fargate handles the infrastructure underneath.

  • Pricing (us-east-1): $0.04048/vCPU/hour + $0.004445/GB/hour; billed per-second, 1-minute minimum
  • Example: 0.5 vCPU + 1 GB ≈ $0.024/hour
  • Minimum CPU (Linux): 0.25 vCPU; scales up to 16 vCPU / 120 GB per task
  • Also available as a launch type for EKS
💡
With Fargate, you pay only for the vCPU and memory your task actually uses — not for idle EC2 capacity. A demo task running 0.25 vCPU + 0.5 GB costs ~$0.006/hour.

AWS Container Services Comparison

Service Reference
1
ECR vs ECS Fargate vs EKS vs App Runner
⌃

All four services are part of the AWS container ecosystem. They serve different layers of the stack and are often used together.

ECR ECS + Fargate EKS App Runner
Purpose Image registry Container orchestration Managed Kubernetes Fully managed container PaaS
Complexity Low Medium High Very Low
Kubernetes needed No No Yes No
Server management N/A AWS manages (Fargate) AWS manages control plane AWS manages everything
Auto-scaling N/A Via CloudWatch Via HPA / KEDA Built-in
Use when Always (store images) Most production workloads Need Kubernetes features Simplest path to production
2025 updates Pull-through cache — Auto Mode, v1.35, Win 2025 VPC support, WAF, custom domains
💡
Start with ECS Fargate for most workloads. Move to EKS when you need Kubernetes-specific features or multi-cloud portability. Use App Runner for the simplest possible deployment.

When to Use What

Decision Guide
1
Container Service Decision Tree
⌃

Use this decision flow to choose the right AWS container service for your use case.

Need to run containers on AWS?
↓
Want the simplest path — no config, just deploy?
App Runner
Need Kubernetes features or multi-cloud portability?
EKS (use EKS Auto Mode to start)
Need flexibility without Kubernetes complexity?
ECS + Fargate
Managing your own EC2 nodes for cost or compliance?
ECS + EC2
⚠️
EKS Auto Mode (launched 2025) makes Kubernetes nearly ops-free — it handles compute, networking, and storage automatically. This narrows the gap between ECS Fargate and EKS significantly for teams that need Kubernetes features.

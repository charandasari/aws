Overview & Architecture

Getting Ready
1
What We Are Building
⌃

We will take a static HTML app, package it in Docker, store it in Amazon ECR, and deploy it to ECS Fargate so it runs as a serverless container accessible from the internet. No EC2 to manage, no SSH, no server patching. The entire workflow — from code on your laptop to a live public URL — takes about 60 minutes following this guide.

Important — Apple Silicon Mac Note
⌃
⚠️
If you are on an Apple Silicon Mac (M1/M2/M3/M4), your Mac builds arm64 Docker images by default. AWS Fargate runs on x86_64 (amd64). You MUST specify --platform linux/amd64 when building images for Fargate, or they will crash with "exec format error".

Always use this build command when targeting Fargate:

AWS Account & IAM User

IAM & Credentials
1
Why Not Use the Root Account
⌃

The root account has unlimited access to everything in your AWS account — including billing, account deletion, and all services. If its credentials are ever compromised, an attacker has full control. It should be locked away after initial account setup and never used for day-to-day work. Create a dedicated IAM user instead.

💡
Best practice: Enable MFA on your root account, then never use it for normal operations.
2
Create an IAM User (Console)
⌃

Navigate: AWS Console → IAM → Users → Create user

  1. User name: containers-demo-user
  2. Check "Provide user access to the AWS Management Console"
  3. Select "I want to create an IAM user"
  4. Set a custom password → uncheck "must create new password at next sign-in"
  5. Click Next
3
Attach Permissions (Console)
⌃

On the permissions page, choose "Attach policies directly" and search for and attach each of these policies:

  • AmazonEC2ContainerRegistryFullAccess — create repos, push/pull images
  • AmazonECS_FullAccess — create clusters, task definitions, services
  • AmazonVPCFullAccess — view and use VPC/subnets/security groups
  • IAMFullAccess — create the task execution role (needed once)
  • CloudWatchLogsFullAccess — view container logs

Click Next → Create user.

✅
For a personal learning account, attaching AdministratorAccess is faster. In a company account, use the specific policies above to follow least-privilege.

Create Access Keys for CLI
⌃
  1. Go to IAM → Users → containers-demo-user
  2. Click the "Security credentials" tab
  3. Click "Create access key"
  4. Use case: Command Line Interface (CLI)
  5. Check the acknowledgment box → Next
  6. Download the .csv file — this is the ONLY time you see the secret key
⚠️
Store the access key CSV safely. If lost, you must create a new one. Never commit access keys to Git — attackers scan GitHub constantly for leaked credentials.

Write the Dockerfile

Docker
1
Project Structure
⌃

Your project needs two things — the HTML file and the Dockerfile. The Dockerfile tells Docker how to package your app. Keep everything inside a containers/ folder.

📁 containers/
📄 Dockerfile      ← instructions for Docker
📁 app/
📄 index.html    ← your web app
2
The Dockerfile
⌃

Three lines are all it takes to containerize a static website. Each line has a specific purpose.

dockerfile
FROM nginx:alpine
COPY app/index.html /usr/share/nginx/html/index.html
EXPOSE 80

Line-by-line explanation:

  • FROM nginx:alpine — Start from the official nginx web server image on Alpine Linux. Alpine is a minimal Linux distribution (~5MB). nginx:alpine is the entire web server compressed into ~22MB — no installation needed.
  • COPY app/index.html /usr/share/nginx/html/index.html — Copy your HTML file into the location where nginx serves files by default. Any file placed here is served over HTTP.
  • EXPOSE 80 — Documents that this container listens on port 80 (HTTP). Does not actually open the port — port mapping happens at runtime.
💡
Only 3 lines. This is the minimum viable Dockerfile for a static website. nginx handles all the web server configuration automatically.
3
The HTML App
⌃

Create the file at app/index.html with this content. It's a dark-themed page that confirms your container is running on Fargate.

html
<!DOCTYPE html>
<html lang="en">
<head>
  <meta charset="UTF-8" />
  <title>Containers on AWS</title>
  <style>
    body {
      font-family: sans-serif;
      background: #0d1117;
      color: #e6edf3;
      display: flex;
      flex-direction: column;
      align-items: center;
      justify-content: center;
      min-height: 100vh;
    }

What is ECR?
⌃

Amazon ECR (Elastic Container Registry) is AWS's private Docker registry. When ECS starts your Fargate task, it pulls the image from ECR — not from your laptop. ECR is private (only your AWS account can access it by default), versioned (each push creates an immutable layer), and integrated natively with ECS.

Authenticate & Push Image

Amazon ECR
1
Authenticate Docker to ECR
⌃

ECR is private. Docker needs a temporary token from AWS to push to it. The token is valid for 12 hours.

bash
aws ecr get-login-password --region $REGION \
  | docker login --username AWS --password-stdin \
    $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com

How the pipe works:

  • aws ecr get-login-password — calls AWS, returns a temporary token string
  • | — pipes that token to the next command
  • docker login --password-stdin — Docker reads the token and authenticates to ECR
Terminal — ~
Login Succeeded
⚠️
If you get a proxy error (proxyconnect tcp: i/o timeout), prepend HTTPS_PROXY="" HTTP_PROXY="" NO_PROXY="*" to the docker push command. This is a Docker Desktop proxy issue.
2
Tag the Image
⌃

Docker uses image names as addresses. By tagging with the full ECR URI, you tell Docker exactly where to send the image when you push.

bash
docker tag containers-demo:latest \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest
3
Push to ECR
⌃
bash
docker push \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest
Terminal — ~/containers
$ docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/containers-demo:latest

Phase 4 — AWS Infrastructure
09

Task Execution Role

AWS Infrastructure
1
What is This Role and Why is it Required?
⌃

When ECS starts your Fargate task, it does not run as you. ECS runs as its own service and needs an IAM role to perform these actions on your behalf:

  • Pull your image from ECR (private registry)
  • Write container logs to CloudWatch

Without this role, the task cannot start. It will either get stuck in PENDING or fail immediately.

💡
This is the #1 most common mistake for first-time ECS users. If your task goes STOPPED immediately with exit code 1 or CannotPullContainerError — check this role first.
You (IAM User)
→
manages
→
ECS Service
→
assumes ecsTaskExecutionRole
→
Pull ECR + Write Logs
2
Check if the Role Already Exists
⌃
bash
aws iam get-role --role-name ecsTaskExecutionRole \
  --query "Role.Arn" --output text
Terminal — ~
# If it exists:
arn:aws:iam::123456789012:role/ecsTaskExecutionRole
 
# If it doesn't exist:
An error occurred (NoSuchEntity)...
3
Create the Role — Console
⌃

Navigate: IAM → Roles → Create role

  1. Trusted entity type: AWS service
  2. Service: Elastic Container Service
  3. Use case: Elastic Container Service Task → Next
  4. Search: AmazonECSTaskExecutionRolePolicy → check the box → Next
  5. Role name: ecsTaskExecutionRole (exact — ECS looks for this name by default)
  6. Create role
✅
The role name must be exactly ecsTaskExecutionRole. ECS uses this as the default. A different name requires manually specifying the ARN in every task definition.
4
Create the Role — CLI (3 commands)
⌃
bash
# Step 1: Write the trust policy — who can use this role
cat > /tmp/ecs-trust.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ecs-tasks.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
EOF

# Step 2: Create the role
aws iam create-role \
  --role-name ecsTaskExecutionRole \
  --assume-role-policy-document file:///tmp/ecs-trust.json

# Step 3: Attach the AWS-managed execution policy
aws iam attach-role-policy \
  --role-name ecsTaskExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

VPC & Security Group

AWS Infrastructure
1
The Default VPC
⌃

Every AWS region comes with a default VPC (Virtual Private Cloud) that has public subnets and an internet gateway pre-configured. This demo uses the default VPC — no custom networking needed.

bash
aws ec2 describe-vpcs \
  --filters Name=isDefault,Values=true \
  --query "Vpcs[0].{VpcId:VpcId,State:State}" \
  --output table
Terminal — ~
--------------------------------------
| DescribeVpcs |
+---------------+--------------------+
| State | VpcId |
+---------------+--------------------+
| available | vpc-0abc12345678 |
+---------------+--------------------+
2
List Public Subnets
⌃

Subnets are subdivisions of the VPC. Public subnets have a route to the internet. We place Fargate tasks in public subnets so they get a public IP.

bash
aws ec2 describe-subnets \
  --filters Name=default-for-az,Values=true \
  --query "Subnets[*].[SubnetId,AvailabilityZone]" \
  --output table
Terminal — ~
+----------------------+---------------+
| subnet-0abc111111 | us-east-1a |
| subnet-0abc222222 | us-east-1b |
| subnet-0abc333333 | us-east-1c |
+----------------------+---------------+
✅
Select at least 2 subnets in different Availability Zones when creating the ECS service — this gives you redundancy.
3
Create Security Group — Console
⌃

Navigate: VPC → Security groups → Create security group

  1. Name: containers-demo-sg
  2. Description: Allow HTTP port 80 for ECS Fargate demo
  3. VPC: select the default VPC
  4. Inbound rules → Add rule: Type=Custom TCP, Port=80, Source=Anywhere-IPv4 (0.0.0.0/0)
  5. Outbound rules: leave default (All traffic allowed)
  6. Create security group
⚠️
Do NOT remove the default outbound rule. Fargate needs outbound internet access to pull the image from ECR and send logs to CloudWatch.
4
Create Security Group — CLI
⌃
bash
# Get the default VPC ID
VPC_ID=$(aws ec2 describe-vpcs \
  --filters Name=isDefault,Values=true \
  --query "Vpcs[0].VpcId" --output text)

# Create the security group
SG_ID=$(aws ec2 create-security-group \
  --group-name containers-demo-sg \
  --description "Allow HTTP port 80 for ECS Fargate" \
  --vpc-id $VPC_ID \
  --query GroupId --output text)

# Allow inbound HTTP on port 80
aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID \
  --protocol tcp --port 80 --cidr 0.0.0.0/0

echo "Security Group ID: $SG_ID"
✅
Save the $SG_ID — you will need it when creating the ECS service.

Why Logging Matters
⌃

When your container runs on Fargate, you have no SSH access. If something goes wrong, CloudWatch Logs is how you see what happened. nginx writes its access and error logs to stdout/stderr, which Fargate streams to CloudWatch automatically — but only if the log group exists and logging is configured in the task definition.

  3. Create

Create Task Definition

ECS Fargate
1
What is a Task Definition?
⌃

A task definition is the blueprint for your container — it defines everything ECS needs to know to run it: which image to use, how much CPU and memory, which ports to open, where to send logs, and which IAM role to use.

Think of it like a declarative docker run command stored in AWS.

FieldValueWhy
Familycontainers-demo-taskName for this task blueprint
Launch typeFargateServerless — no EC2
CPU512 (0.5 vCPU)Minimum for a simple web app
Memory1024 (1 GB)Minimum recommended for nginx
Execution roleecsTaskExecutionRoleAllows ECS to pull from ECR + write logs
Container imageECR URI:latestThe image we pushed in Step 8
Container port80HTTP traffic
Log driverawslogsStream logs to CloudWatch
2
Create Task Definition — Console
⌃

Navigate: ECS → Task Definitions → Create new task definition

  1. Task definition family: containers-demo-task
  2. Launch type: AWS Fargate
  3. Task execution role: ecsTaskExecutionRole
  4. CPU: 0.5 vCPU
  5. Memory: 1 GB
  6. Click "Add container":
    • Container name: web
    • Image URI: paste your ECR URI (123456789012.dkr.ecr.us-east-1.amazonaws.com/containers-demo:latest)
    • Port mappings: 80 / TCP
  7. Expand "Logging":
    • Log driver: awslogs
    • awslogs-group: /ecs/containers-demo-task
    • awslogs-region: us-east-1
    • awslogs-stream-prefix: ecs
  8. Click Create
⚠️
If Task execution role is blank or set to None, ECS cannot pull your image from ECR. Select ecsTaskExecutionRole from the dropdown.
3
Create Task Definition — CLI
⌃
bash
aws ecs register-task-definition \
  --family containers-demo-task \
  --network-mode awsvpc \
  --requires-compatibilities FARGATE \
  --cpu "512" \
  --memory "1024" \
  --execution-role-arn arn:aws:iam::$AWS_ACCOUNT_ID:role/ecsTaskExecutionRole \
  --region $REGION \
  --container-definitions '[{
    "name": "web",
    "image": "'"$AWS_ACCOUNT_ID"'.dkr.ecr.'"$REGION"'.amazonaws.com/containers-demo:latest",
    "portMappings": [{"containerPort": 80, "protocol": "tcp"}],
    "essential": true,
    "logConfiguration": {
      "logDriver": "awslogs",
      "options": {
        "awslogs-group": "/ecs/containers-demo-task",
        "awslogs-region": "'"$REGION"'",
        "awslogs-stream-prefix": "ecs"
      }
    }
  }]' \
  --query "taskDefinition.{Family:family,Revision:revision,Status:status}" \
  --output table

Create ECS Service

ECS Fargate
1
What is an ECS Service?
⌃

A service is the runtime manager. It keeps your desired number of tasks running at all times, restarts tasks automatically if they crash, handles rolling deployments when you update the image, and can auto-scale based on CPU or memory.

For this demo we set desired count = 1 (one running container).

2
Get Your Subnet IDs
⌃

Before creating the service, get the subnet IDs to use in the next step:

bash
aws ec2 describe-subnets \
  --filters Name=default-for-az,Values=true \
  --query "Subnets[*].SubnetId" \
  --output text

Copy at least 2 subnet IDs from the output.

3
Create Service — Console
⌃

Navigate: ECS → Clusters → containers-demo-cluster → Services tab → Create

  1. Launch type: Fargate
  2. Task definition: containers-demo-task (Revision: 1)
  3. Service name: containers-demo-service
  4. Desired tasks: 1

1
Why Clean Up
⌃

A Fargate task running 24/7 costs ~$0.58/day (0.5vCPU + 1GB). ECR stores images at $0.10/GB/month. Clean up after testing to avoid charges.

ECR image storage$0.10/GB/monthsame
ECS cluster (no tasks)$0.00$0.00
2
Delete in the Correct Order
⌃
⚠️
AWS has dependency rules. Delete in this exact order or you will get errors (e.g. you cannot delete a cluster that has active services).
  • 1. Scale ECS service to 0 (drain tasks)
  • 2. Delete ECS service
  • 3. Deregister task definition
  • 4. Delete ECS cluster
  • 5. Delete ECR repository (--force)
  • 6. Delete CloudWatch log group
  • 7. Delete security group
  • 8. Detach IAM policy from role
  • 9. Delete IAM role
3
Full Cleanup — CLI
⌃
bash
# 1. Scale service to 0
aws ecs update-service \
  --cluster containers-demo-cluster \
  --service containers-demo-service \
  --desired-count 0 --region $REGION

# 2. Delete ECS service
aws ecs delete-service \
  --cluster containers-demo-cluster \
  --service containers-demo-service --region $REGION

# 3. Deregister task definition
aws ecs deregister-task-definition \
  --task-definition containers-demo-task:1 --region $REGION

# 4. Delete ECS cluster
aws ecs delete-cluster \
  --cluster containers-demo-cluster --region $REGION

# 5. Delete ECR repo + all images
aws ecr delete-repository \
  --repository-name containers-demo \
  --force --region $REGION

# 6. Delete CloudWatch log group
aws logs delete-log-group \
  --log-group-name /ecs/containers-demo-task --region $REGION

# 7. Delete security group
aws ec2 delete-security-group \
  --group-id $SG_ID --region $REGION

# 8. Detach policy from IAM role
aws iam detach-role-policy \
  --role-name ecsTaskExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

# 9. Delete IAM role
aws iam delete-role --role-name ecsTaskExecutionRole

Cleanup — Console Order
⌃
  1. ECS → Services → select service → Update → Desired tasks = 0 → Save
  2. ECS → Services → select service → Delete
  3. ECS → Task Definitions → select → Deregister
  4. ECS → Clusters → select → Delete cluster
  5. ECR → Repositories → containers-demo → Delete
  6. CloudWatch → Log groups → /ecs/containers-demo-task → Delete
  7. VPC → Security groups → containers-demo-sg → Delete
  8. IAM → Roles → ecsTaskExecutionRole → Detach policy → Delete role

What Are Containers?

Foundations · 12 min
1
The Problem: "Works on My Machine"
⌃

You write code on your laptop. It runs perfectly. You hand it to a colleague — it crashes. You deploy to production — it crashes differently. This is the most common frustration in software development, and it has one root cause: environmental differences.

  • Your laptop has Python 3.11 — the server has Python 3.8
  • You installed library version 2.3.1 — production has 1.9.0
  • Your OS is macOS — the server is Linux, with different file paths and system calls
  • Environment variables, config files, SSL certificates — all differ

Containers solve this entirely. A container packages your application code together with every dependency it needs — the exact Python version, the exact library versions, the config files, the runtime — into a single portable unit. If it runs in a container on your laptop, it will run identically in a container on AWS.

✅
Real-world impact: Netflix deploys thousands of independent microservices daily — each in its own container. Teams ship faster because environment parity is guaranteed. A bug reproducible in production can be reproduced exactly on any developer's laptop.
2
VMs vs Containers — Apartment vs Co-Working Desk
⌃

Before containers, the standard solution was Virtual Machines (VMs). A VM gives each app its own complete computer — its own operating system, its own kernel, its own everything. Think of it like renting a full apartment: you own the space entirely, but it takes time to move in, costs a lot, and you're paying for rooms you never use.

A container is more like a co-working desk. You share the building's infrastructure (the OS kernel) with others, but your desk is yours — your files, your tools, your environment. Move in instantly. Pay only for what you use.

Dimension Virtual Machine Container
Startup time 1–5 minutes < 1 second
Image size 1–20 GB 10–200 MB
OS per instance Full OS per VM Shared host kernel
Isolation Strong (hardware-level) Good (process-level)
Portability Hypervisor-dependent Runs anywhere Docker runs
Resource overhead High — full OS per VM Minimal — shared kernel
Density ~10s per host 100s per host
💡
VMs and containers are not mutually exclusive. In AWS, Fargate actually runs your containers inside a micro-VM (using Firecracker) to give you both the speed of containers and the isolation of VMs. You get the best of both worlds without managing either.
3
The Shipping Container Analogy
⌃

In 1956, a truck driver named Malcolm McLean had a simple idea: instead of loading cargo piece by piece onto ships, what if everything went into standardized metal boxes? Longshoremen wouldn't need to know what was inside. Cranes could load any box onto any ship. Ships could dock anywhere in the world.

Shipping costs dropped by 90%. Global trade transformed. The standardized shipping container is now one of the most important inventions of the 20th century.

Docker did the same thing for software in 2013. Before Docker, deploying software meant hand-crafting server configurations for each environment. With Docker, you package your app into a standardized container image. That image runs identically everywhere:

MacBook Pro
→
CI/CD Server
→
AWS Fargate
→
GCP · Azure

The same container image — no changes, no reconfiguration. The runtime handles the rest. This is why Docker's tagline is "Build once, run anywhere."

✅
The Open Container Initiative (OCI) now standardizes container formats, so Docker images work with containerd, Podman, and other runtimes. You're not locked into Docker — the standard is open.
4
Images vs Containers — Recipe vs Meal
⌃

One of the most important distinctions in container terminology: an image is not a container. They are related but different things.

  • Image — a read-only blueprint. Like a recipe card. It defines what the container will contain: which files, which software, which configuration. You can share it, version it, store it in a registry.
  • Container — a running instance of an image. Like the meal you cooked from the recipe. It's alive, it uses CPU and memory, it can write files. You can run 10 containers from the same image simultaneously.
Dockerfile (instructions)
↓ docker build
Image (static blueprint)
↓ docker run
Container (running process)

You build an image once and store it. You run containers from that image as many times as needed. When a container crashes, ECS simply starts a new one from the same image — instantly, identically.

Docker — The Packaging Tool

Foundations · 15 min
1
What Docker Is
⌃

Docker is not a single thing — it's a toolkit with three distinct components that work together:

  • Docker Engine — the daemon that actually runs containers on your machine. It manages the container lifecycle: start, stop, pause, kill. Runs as a background service.
  • Docker CLI — the command-line tool you interact with. Commands like docker build, docker run, docker push. The CLI talks to the Engine.
  • Docker Hub — a public registry of pre-built images. Over 8 million images. When you write FROM nginx:alpine in a Dockerfile, Docker pulls the official nginx image from Hub automatically — no installation needed.
💡
In production you'll use Amazon ECR instead of Docker Hub — your own private registry inside AWS. ECR images never leave your AWS account, load faster from ECS (same region, no internet hop), and are scanned for vulnerabilities automatically.
2
The Dockerfile — Our 3-Line Example
⌃

A Dockerfile is a plain text file that defines how to build your container image. It's a sequence of instructions — each instruction becomes a layer. Here's the complete Dockerfile for this demo:

dockerfile 
FROM nginx:alpine
COPY app/index.html /usr/share/nginx/html/index.html
EXPOSE 80

Three lines. A complete, production-deployable web server. Here's what each instruction does:

InstructionWhat it doesLayer size
FROM nginx:alpine Start from the official nginx web server image, based on Alpine Linux (a 5MB minimal Linux distro). Pulls from Docker Hub. This gives you a fully configured nginx with zero setup. ~22 MB
COPY app/index.html ... Copy our HTML file from the local build context into the nginx serving directory inside the image. nginx serves everything in /usr/share/nginx/html/. ~2 KB
EXPOSE 80 Documents that this container listens on port 80. This is metadata — it doesn't actually open the port. Port mapping happens at runtime with -p 80:80 or via ECS task definition. 0 B
3

4
Layer Caching — Why Pushes Are Fast After the First
⌃

Every instruction in a Dockerfile creates a layer — an immutable snapshot of the filesystem at that point. Docker caches layers. When you rebuild an image, Docker checks: has anything changed for this instruction? If not, it reuses the cached layer instead of rebuilding.

In our 3-line Dockerfile, only one layer ever changes: the COPY layer that contains your HTML file. The FROM nginx:alpine layer (22 MB) is cached forever.

docker push — second push after editing index.html
$ docker push 123456789.dkr.ecr.us-east-1.amazonaws.com/my-app:v2
The push refers to repository [123456789.dkr.ecr.us-east-1.amazonaws.com/my-app]
html-layer: Pushing 2.048kB
html-layer: Pushed
nginx-layer: Layer already exists
alpine-layer: Layer already exists
v2: digest: sha256:a1b2c3... size: 527

Only the 2 KB HTML layer re-uploads. The 22 MB nginx base layer is already in ECR — it never re-uploads. This is why subsequent pushes take seconds, not minutes.

✅
Optimization tip: In larger Dockerfiles, put instructions that change frequently (like COPY . .) at the bottom, and stable instructions (like RUN apt-get install) at the top. This maximizes cache hits and keeps build times fast.
5
CRITICAL: Apple Silicon Warning
⌃
⚠️
Apple Silicon (M1/M2/M3/M4) users — read this before building anything.

Macs with Apple Silicon use the arm64 CPU architecture. AWS Fargate (by default) runs on linux/amd64 (x86_64). If you build an image on your Mac without specifying a platform, Docker builds an arm64 image. When Fargate tries to run it, you get a cryptic error: exec format error — the container crashes immediately on launch.

Always add --platform linux/amd64 to your build command:
bash 
docker build --platform linux/amd64 -t my-app:v1 .

Docker Desktop on Mac uses QEMU emulation to build amd64 images transparently. It's slightly slower than native arm64 builds, but the resulting image is fully compatible with Fargate. Alternatively, configure Fargate to use ARM64 architecture in the task definition — but then you must build arm64 images consistently.

Why ECS + Fargate — Who Uses It at Scale
⌃

The core value proposition of Fargate is zero infrastructure management. With EC2-based ECS, you provision and patch a fleet of EC2 instances, manage their capacity, handle node failures. With Fargate, you describe what you want to run (CPU, memory, image) and AWS handles every layer beneath that.

  • No EC2 instances to provision, patch, or right-size
  • No cluster capacity planning — Fargate scales instantly to any workload
  • Pay per second — a task running for 30 seconds costs 30 seconds of compute
  • Scales from 1 task to thousands without any configuration changes
✅
Real-world adoption: Duolingo runs all 500+ microservices on ECS Fargate. Lyft migrated from Kubernetes to ECS Fargate to reduce operational overhead. Samsung uses ECS at global scale. AWS itself runs many internal services on Fargate.

Amazon ECR — Your Image Registry

AWS Services · 10 min
1
What ECR Is
⌃

Amazon Elastic Container Registry (ECR) is GitHub for Docker images — a fully managed service for storing, versioning, and distributing container images. Every image you build gets pushed to ECR, and ECS pulls from ECR when launching tasks.

ECR organizes images into repositories — one repository per application. Each push creates a new image tag (typically a version like v1, v2, or a git commit SHA). ECS always pulls the specific tag you specify in the task definition.

Key built-in features:

  • Image scanning — automatic vulnerability scanning using Amazon Inspector on push. Finds known CVEs in your image layers.
  • Lifecycle policies — automatically delete images older than N days or keep only the last N versions. Prevents runaway storage costs.
  • Cross-region replication — replicate images to multiple regions automatically for multi-region deployments.
  • Immutable tags — optionally prevent overwriting existing tags. Enforces version discipline.
2
ECR vs Docker Hub
⌃

Docker Hub is the public default. ECR is the right choice for any professional AWS workload.

DimensionDocker HubAmazon ECR

ECS Integration Manual credentials setup required Native — zero config, same-account pull
Pull latency from ECS Internet round-trip Same AWS region — milliseconds
Cost Free tier limited; paid plans for teams $0.10/GB/month storage; data transfer free within region
Security scanning Basic (paid tier) Amazon Inspector (enhanced scanning)
3
Pricing — Nearly Free for This Demo
⌃

ECR pricing is simple: $0.10 per GB per month for storage. Data transfer within the same AWS region (e.g., ECR to ECS in us-east-1) is free.

  • Our nginx:alpine image: ~22 MB total
  • Monthly storage cost: 0.022 GB × $0.10 = $0.0022/month — effectively zero
  • Free tier: 500 MB free for your first 12 months
  • Even at 50 image versions: 50 × 22 MB = 1.1 GB = $0.11/month
💡
Set up a lifecycle policy to auto-delete untagged images and keep only the last 10 tagged versions. This keeps storage costs near zero even on active repositories with frequent deployments.
4
The Push Workflow — 3 Commands
⌃

Pushing an image to ECR always follows the same three-step sequence: authenticate, tag, push. You do this every time you want to deploy a new version.

bash 
# Step 1: Authenticate Docker to ECR (token valid 12 hours)
aws ecr get-login-password --region us-east-1 \
  | docker login --username AWS --password-stdin \
    123456789012.dkr.ecr.us-east-1.amazonaws.com

# Step 2: Tag your local image with the ECR URI
docker tag my-app:v1 \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1

# Step 3: Push to ECR
docker push \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/my-app:v1

Replace 123456789012 with your AWS Account ID and us-east-1 with your region. After the push completes, copy the full image URI — you'll need it when creating the ECS task definition.

Amazon ECS — The Orchestrator

AWS Services · 15 min
1
What ECS Does
⌃

Amazon Elastic Container Service (ECS) is a container orchestrator — it manages the lifecycle of your containers at scale. You tell ECS what you want, and it makes it happen.

Think of ECS as a hotel manager. You tell the manager: "Keep 3 rooms occupied at all times." The manager doesn't care which specific rooms — that's their problem. If a guest (container) checks out unexpectedly (crash), the manager immediately fills the room (restarts a container). If you say "I need 10 rooms now" (scale out), the manager arranges it without you touching anything.

ECS handles:

  • Where to run each container (which Fargate instance, which AZ)
  • How many copies to run (desired count)
  • What to do when a container crashes (automatic restart)
  • How to deploy new versions (rolling update, blue/green)
  • Health checking — replace unhealthy containers
  • Load balancer integration — register/deregister tasks automatically
2
Core Concepts — The Hotel Analogy
⌃

ECS has five core primitives. Understanding these is essential before you can configure anything.

ECS ConceptHotel AnalogyWhat it actually is
Cluster The hotel building A logical grouping for your services and tasks. In Fargate mode, it's just a namespace — no servers to manage.
Service The hotel manager Maintains a specified number of running tasks. Restarts tasks on failure. Handles deployments and scaling.
Task Definition The room specification The blueprint: which container image, how much CPU/memory, which ports, which IAM role, environment variables. Versioned (revision 1, revision 2, etc.).
Task An occupied room A running instance of a task definition. One task = one set of containers. A service runs multiple tasks for redundancy.
Container The guest The actual Docker container running inside a task. One task can have multiple containers (sidecar pattern).
💡
Task Definition revisions: Every time you update a task definition, ECS creates a new revision (myapp:1, myapp:2, myapp:3...). Old revisions remain available. To roll back a deployment, update the service to use an older revision — instantly.
3
Deployment Strategies
⌃

ECS supports three deployment strategies. Each trades off simplicity against control and risk tolerance.

StrategyHow it worksZero downtimeUsed by
Rolling Update Gradually replace old tasks with new ones. Default strategy. Configurable min/max percentages. Yes Most teams
Blue/Green Deploy new version alongside old (two environments). Switch traffic when new version passes health checks. Instant rollback. Yes Amazon.com, enterprises
Canary Send a small % (e.g., 10%) of traffic to new version first. Validate. Then shift remaining 90%. Yes Netflix, risk-sensitive teams
4
Rolling Update — Zero Downtime by Default
⌃

When you update a service with the default rolling deployment, ECS uses two parameters to maintain availability:

  • Minimum healthy percent (default 100%) — ECS will never let your running task count drop below this percentage of desired count. At 100%, it can't kill any old tasks until new ones are running.
  • Maximum percent (default 200%) — ECS can run up to this many tasks during deployment. At 200%, it can double the task count temporarily.

Result: with defaults (min 100%, max 200%), if you have 2 running tasks, ECS starts 2 new tasks (4 total), confirms they're healthy, then terminates the 2 old tasks. Your app stays live throughout. Users never see downtime.

✅
For this demo with 1 desired task, ECS will start the new task, wait for it to pass health checks, then stop the old one. There's a brief moment with both running. Always works — even with a single task.

AWS Fargate — Serverless Compute

AWS Services · 12 min
1
What Fargate Is — The Cloud Kitchen Analogy
⌃

Imagine you want to open a restaurant. Option A: buy a building, install the kitchen equipment, hire maintenance staff, manage the lease, upgrade appliances when they break. Option B: use a cloud kitchen — send your recipes, they handle the kitchen. You pay per meal cooked.

EC2-based ECS is Option A. You provision and manage EC2 instances (the kitchen). You patch them, right-size them, manage cluster capacity. When a node fails, you replace it.

AWS Fargate is Option B. You provide the container image (the recipe) and a task definition (serving size, dietary requirements). AWS handles all the compute infrastructure beneath it. There are no EC2 instances visible to you — just tasks running.

What Fargate manages so you don't have to:

  • Provisioning the underlying compute instances
  • OS patching and security updates
  • Right-sizing — you specify CPU/memory per task, not instance type
  • Bin-packing (fitting containers onto hosts efficiently)
  • Node failure recovery
  • Capacity planning — Fargate scales to any demand instantly
2
Fargate Pricing — Real Numbers
⌃

Fargate charges for vCPU and memory separately, per second. There are no upfront costs, no minimum commitments. You pay only for the time your tasks are running.

3
Task Lifecycle — Every State Explained
⌃

When you launch or stop a Fargate task, it transitions through five states. Understanding these helps you diagnose stuck or failing tasks.

PROVISIONING
AWS is allocating compute resources (a Firecracker microVM). Typically takes 5–30 seconds. Network interfaces are being attached, IAM credentials are being vended.
PENDING
Resources allocated. Fargate is pulling the Docker image from ECR. If your image is large or ECR auth fails, you can get stuck here. Watch for CannotPullContainerError.
RUNNING
Your container is alive and processing requests. The task stays here until you stop it, it crashes, or ECS replaces it during a deployment. This is the healthy steady state.
STOPPING
ECS sent SIGTERM to the container. Default stop timeout: 30 seconds. If your app doesn't shut down in time, SIGKILL is sent. Configure the stop timeout in the task definition for graceful shutdown.
STOPPED
Task has exited. ECS records the stop reason and exit code. Visible in the ECS console for up to 1 hour. If the service's desired count requires it, ECS immediately starts a replacement.
4
Fargate vs EC2 Launch Type — When to Use Each
⌃
💡
Choose Fargate when:
  • You don't want to manage EC2 instances
  • Workloads are variable or unpredictable
  • Tasks run intermittently (batch jobs, event-driven workloads)
  • Team is small — operational simplicity matters
  • You're just getting started with containers
Choose EC2 launch type when:
  • You need GPU instances (Fargate doesn't support GPU)
  • Very high steady-state workloads where Reserved Instances save cost
  • You need specific instance types or custom AMIs
  • Windows containers (Fargate supports Windows but with limitations)
  • Strict network requirements (e.g., placement groups)

IAM — Permissions & Roles

AWS Services · 15 min
1
Two Roles — Which Does What
⌃

ECS Fargate uses two distinct IAM roles that are often confused. Getting this wrong is the most common reason deployments fail.

RoleWho uses itRequired?What it grants
ecsTaskExecutionRole The ECS/Fargate infrastructure itself Always required Pull images from ECR, write logs to CloudWatch
Task Role Your application code inside the container Only if app needs AWS access Read S3, query DynamoDB, invoke Lambda, etc.

For this demo, our nginx container just serves static HTML. It never calls any AWS API. So we need the execution role (always required) but not a task role.

If you were building an app that reads from S3 or writes to DynamoDB, you'd add a task role with those specific permissions. The app gets AWS credentials automatically via the container metadata endpoint — no secrets in the code.

2
What ecsTaskExecutionRole Grants — and Why It's #1 Mistake
⌃

Without this role, ECS cannot pull your image from ECR and cannot write logs to CloudWatch. Tasks will fail to start with a cryptic permissions error. This is the single most common ECS mistake for people new to the service.

The role needs these exact permissions:

📄 AmazonECSTaskExecutionRolePolicy (AWS Managed)
# ECR permissions — pull your container image "ecr:GetAuthorizationToken" # get ECR auth token "ecr:BatchCheckLayerAvailability" # check which layers exist "ecr:GetDownloadUrlForLayer" # get presigned download URLs "ecr:BatchGetImage" # pull the image manifest # CloudWatch Logs permissions — write container logs "logs:CreateLogStream" # create a log stream in your group "logs:PutLogEvents" # write log lines
⚠️
If your task won't start and you see "CannotPullContainerError" or "AccessDeniedException", the execution role is the first thing to check. Verify it exists, verify ECS trusts it, verify it has the managed policy attached.
3
Create the Execution Role — 3 Commands
⌃

Creating the role requires three AWS CLI commands. First you create a trust policy document that allows ECS to assume the role, then you create the role, then you attach the managed policy.

bash 
# Step 1: Create the trust policy document
cat > ecs-trust-policy.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ecs-tasks.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
EOF

# Step 2: Create the IAM role
aws iam create-role \
  --role-name ecsTaskExecutionRole \
  --assume-role-policy-document file://ecs-trust-policy.json

# Step 3: Attach the AWS managed policy
aws iam attach-role-policy \
  --role-name ecsTaskExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy
✅
You only need to create this role once per AWS account. Once created, all ECS task definitions in any cluster in that account can reference it. You don't re-create it for each service.
4
Verify the Role Exists
⌃

After creating the role, verify it exists and grab its ARN — you'll need the ARN when configuring the task definition.

bash 
aws iam get-role \
  --role-name ecsTaskExecutionRole \
  --query "Role.Arn" \
  --output text
Expected output
arn:aws:iam::123456789012:role/ecsTaskExecutionRole

Copy this ARN. You will paste it into the task definition as the executionRoleArn value. Without it, ECS will reject the task definition or fail to launch tasks.

VPC & Networking

AWS Services · 12 min
1
VPC — Your Private Office Building
⌃

A Virtual Private Cloud (VPC) is your private, isolated network inside AWS. Think of it as a private office building your company rents. The building exists inside a larger complex (AWS's network), but only people with your building's access cards can get in.

Inside the VPC, everything maps to a real-world analogy:

VPC ComponentBuilding AnalogyWhat it does
VPC The office building Private IP address space (e.g., 10.0.0.0/16) isolated from other AWS customers
Subnet A floor of the building Subdivides the VPC. Public subnets have internet access; private subnets don't.
Security Group Access card reader Stateful firewall. Controls which traffic can reach your container (inbound) and what it can reach (outbound).
Internet Gateway The building's main entrance Allows traffic to flow between your VPC and the public internet.
Fargate Task An employee at their desk Gets a private IP from the subnet. If in a public subnet with public IP enabled, reachable from the internet.
2
The Default VPC — Already Ready to Use
⌃

AWS creates a default VPC in every region for every account. It comes pre-configured with public subnets in each Availability Zone and an internet gateway already attached. You don't need to create anything for this demo.

Verify your default VPC exists and grab its ID:

bash 
# Get default VPC ID
aws ec2 describe-vpcs \
  --filters "Name=isDefault,Values=true" \
  --query "Vpcs[0].VpcId" \
  --output text

# Get public subnet IDs in the default VPC
aws ec2 describe-subnets \
  --filters "Name=defaultForAz,Values=true" \
  --query "Subnets[*].SubnetId" \
  --output text
Example output
vpc-0a1b2c3d4e5f67890
subnet-0abc1234 subnet-0def5678 subnet-0ghi9012

Save these IDs. You'll use 2–3 subnet IDs when creating the ECS service — spreading tasks across multiple subnets means they run in different Availability Zones, giving you automatic resilience.

3
Security Groups — Your Container's Firewall
⌃

A security group is a stateful firewall that controls what traffic can reach your Fargate task. "Stateful" means if you allow inbound traffic on port 80, the response traffic is automatically allowed out — you don't need a separate outbound rule for responses.

For this demo, you need one security group with two rules:

  • Inbound: TCP port 80 from anywhere (0.0.0.0/0) — lets users access your app
  • Outbound: All traffic to anywhere — required for Fargate to pull from ECR and write to CloudWatch
bash 
# Create security group
aws ec2 create-security-group \
  --group-name ecs-fargate-sg \
  --description "Allow HTTP inbound for ECS Fargate" \
  --vpc-id vpc-0a1b2c3d4e5f67890

# Add inbound rule: TCP 80 from internet
aws ec2 authorize-security-group-ingress \
  --group-id sg-0123456789abcdef0 \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0
⚠️
Never remove the outbound "all traffic" rule from a Fargate security group. Fargate tasks must be able to reach ECR (to pull images) and CloudWatch Logs (to write logs) over HTTPS. Blocking outbound traffic breaks both. This is a common misconfiguration.
4
Public IP vs ALB — Dev vs Production
⌃
💡
Public IP (what we use in this demo):
Each Fargate task gets its own public IP address when launched in a public subnet with "Auto-assign public IP" enabled. Simple — no extra cost, works immediately. Two downsides: the IP changes on every task restart or redeployment, and you can't use HTTPS without extra work. Use this for development and demos only.

Application Load Balancer (production):
An ALB sits in front of your ECS service and gives you a stable DNS name (e.g., my-app-123456.us-east-1.elb.amazonaws.com). The ALB forwards requests to your tasks; ECS automatically registers and deregisters tasks as they scale. You get HTTPS termination, health checks, multi-AZ routing, and path-based routing — all at ~$16/month base cost. Use this for any production workload.

Local Setup & Docker Build

Start Here · 10 min
1
Project Structure
⌃

Your project needs two things — the HTML file and the Dockerfile. Keep them in this structure:

📁 containers/
📄 Dockerfile ← build instructions
📁 app/
📄 index.html ← your web app
✅
Always test locally before pushing to AWS. Local iteration is free and takes seconds. AWS iteration costs money and takes 30–60 seconds per cycle.
2
Build for the Right Architecture
⌃

Apple Silicon Mac (M1/M2/M3/M4) — use this command:

bash
docker buildx build --platform linux/amd64 -t containers-demo:latest --load .

Intel Mac / Linux — standard build:

bash
docker build -t containers-demo:latest .
⚠️
Apple Silicon builds arm64 by default. Fargate runs x86_64 (amd64). Mixing causes exec format error — the container crashes immediately on startup with exit code 255. Always use --platform linux/amd64.
3
Test Locally Before Pushing
⌃

Always verify the container works on your laptop before pushing to AWS:

bash
# Run the container
docker run -d -p 8080:80 --name demo-local containers-demo:latest

# Verify HTTP 200
curl -s -o /dev/null -w "HTTP %{http_code}" http://localhost:8080

# Stop and clean up
docker stop demo-local && docker rm demo-local
Terminal
$ curl -s -o /dev/null -w "HTTP %{http_code}" http://localhost:8080
HTTP 200 ← container is healthy and serving

Push Image to ECR

Upload to AWS · 5 min
1
Create ECR Repository
⌃

Create a private repository to store your image. You only do this once per app.

bash
aws ecr create-repository \
  --repository-name containers-demo \
  --region us-east-1 \
  --query "repository.repositoryUri" --output text
Terminal
123456789012.dkr.ecr.us-east-1.amazonaws.com/containers-demo
2
Authenticate, Tag & Push
⌃

Three commands — authenticate Docker to ECR, tag your image with the ECR address, then upload it:

bash
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGION=us-east-1

# 1. Authenticate Docker to ECR (token valid 12 hours)
aws ecr get-login-password --region $REGION \
  | docker login --username AWS --password-stdin \
    $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com

# 2. Tag with ECR address
docker tag containers-demo:latest \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest

# 3. Push (proxy bypass if Docker Desktop proxy issue)
HTTPS_PROXY="" HTTP_PROXY="" NO_PROXY="*" \
  docker push $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest
⚠️
If you see proxyconnect tcp: i/o timeout — Docker Desktop has an internal proxy blocking ECR. Prepend HTTPS_PROXY="" HTTP_PROXY="" NO_PROXY="*" to the push command to bypass it.

Create AWS Infrastructure

One-Time Setup · 5 min
1
Everything You Need — Checklist
⌃

Create these resources once. They persist across all your deployments.

  • ecsTaskExecutionRole — IAM role that lets ECS pull from ECR and write logs
  • ECR repository — private image registry
  • ECS cluster — logical grouping for your services
  • Security group — firewall allowing port 80 inbound
  • CloudWatch log group — where container logs land
2
Create All Infrastructure — CLI
⌃
bash
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGION=us-east-1

# IAM Task Execution Role
cat > /tmp/ecs-trust.json << EOF
{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":{"Service":"ecs-tasks.amazonaws.com"},"Action":"sts:AssumeRole"}]}
EOF

  --assume-role-policy-document file:///tmp/ecs-trust.json
aws iam attach-role-policy --role-name ecsTaskExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy

# ECS Cluster
aws ecs create-cluster --cluster-name containers-demo-cluster --region $REGION

# Security Group (port 80 inbound)
VPC_ID=$(aws ec2 describe-vpcs --filters Name=isDefault,Values=true \
  --query "Vpcs[0].VpcId" --output text)
SG_ID=$(aws ec2 create-security-group --group-name containers-demo-sg \
  --description "HTTP 80" --vpc-id $VPC_ID --query GroupId --output text)
aws ec2 authorize-security-group-ingress --group-id $SG_ID \
  --protocol tcp --port 80 --cidr 0.0.0.0/0
echo "SG: $SG_ID"

# CloudWatch Log Group
aws logs create-log-group --log-group-name /ecs/containers-demo-task --region $REGION

Deploy to ECS Fargate

Launch · 2 min
1
Create the ECS Service
⌃

The service is the long-running manager that keeps your desired number of tasks running. Get your subnet IDs first, then create the service:

bash
aws ecs create-service \
  --cluster containers-demo-cluster \
  --service-name containers-demo-service \
  --task-definition containers-demo-task:1 \

✅
assignPublicIp=ENABLED is required for public access. Without it the task has no public IP and is unreachable from the internet.
2
Watch the Task Start
⌃

Poll the task status — it moves through three states before serving traffic:

Terminal — task lifecycle
$ aws ecs describe-tasks ... --query "tasks[0].lastStatus"
"PROVISIONING" ← AWS allocating Fargate compute (~10s)
"PENDING" ← pulling image from ECR (~15-30s)
"RUNNING" ← container is live and serving ✓

Verify & Access Your App

Go Live
1
Get the Public IP
⌃

Fargate assigns a public IP via an Elastic Network Interface (ENI). Three API calls to retrieve it:

bash
TASK_ARN=$(aws ecs list-tasks \
  --cluster containers-demo-cluster \
  --service-name containers-demo-service \
  --query "taskArns[0]" --output text)

ENI_ID=$(aws ecs describe-tasks \
  --cluster containers-demo-cluster --tasks $TASK_ARN \
  --query "tasks[0].attachments[0].details[?name=='networkInterfaceId'].value" \
  --output text)

PUBLIC_IP=$(aws ec2 describe-network-interfaces \
  --network-interface-ids $ENI_ID \
  --query "NetworkInterfaces[0].Association.PublicIp" --output text)

echo "App live at: http://$PUBLIC_IP"
💡
Why 3 API calls? Fargate doesn't expose the IP directly on the task. The IP lives on the ENI (network card) that AWS attaches to the task. You follow the chain: task → ENI ID → public IP.
2
IP Changes on Every Redeploy
⌃

Every time ECS replaces a Fargate task (redeploy, crash restart, scale event), the new task gets a new ENI and a new public IP. The old IP immediately becomes unreachable.

⚠️
This is why you need an ALB in production. An Application Load Balancer has a fixed DNS name (myapp.us-east-1.elb.amazonaws.com) that never changes regardless of how many tasks come and go behind it. Direct public IPs are for dev/testing only.
3
View Container Logs
⌃

Since you have no SSH access to Fargate, CloudWatch Logs is your window into the container. View logs in real time:

bash
aws logs tail /ecs/containers-demo-task --follow --region us-east-1
✅
If a task stops immediately after starting, check logs first: aws logs tail /ecs/containers-demo-task --region us-east-1. The error is almost always there — bad image URI, wrong port, missing file, or architecture mismatch.

Making & Deploying Changes

Operations
1
The Change → Deploy Workflow
⌃

Every code change follows the same 4-step loop:

Edit index.html
→
docker build (--no-cache)
→
docker push to ECR
→
ECS update-service

Wait ~30 seconds, then refresh the browser. ECS performs a rolling replacement — new task starts, traffic shifts, old task drains and stops. Zero downtime.

2
Full Redeploy Script
⌃

Save this as redeploy.sh and run it after every change. It handles build, push, and deployment atomically — pinning to the exact image digest to avoid ECS caching issues:

bash
#!/bin/bash
set -e

AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGION=us-east-1
ECR_URI=$AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo

# 1. Build fresh — --no-cache prevents stale layer caching
docker buildx build --no-cache --platform linux/amd64 -t containers-demo:latest --load .

# 2. Tag and push to ECR
docker tag containers-demo:latest $ECR_URI:latest
HTTPS_PROXY="" HTTP_PROXY="" NO_PROXY="*" docker push $ECR_URI:latest

# 3. Get the exact image digest (avoids :latest caching issues)
DIGEST=$(aws ecr describe-images --repository-name containers-demo \
  --image-ids imageTag=latest \
  --query "imageDetails[0].imageDigest" --output text)

# 4. Register new task definition pinned to exact digest
aws ecs register-task-definition --family containers-demo-task \
  --network-mode awsvpc --requires-compatibilities FARGATE \
  --cpu "512" --memory "1024" \
  --execution-role-arn arn:aws:iam::$AWS_ACCOUNT_ID:role/ecsTaskExecutionRole \
  --container-definitions "[{\"name\":\"web\",\"image\":\"$ECR_URI@$DIGEST\",...}]"

# 5. Get new revision and deploy
REVISION=$(aws ecs describe-task-definition --task-definition containers-demo-task \
  --query "taskDefinition.revision" --output text)
aws ecs update-service --cluster containers-demo-cluster \
  --service containers-demo-service \
  --task-definition containers-demo-task:$REVISION

echo "Deployed revision $REVISION — live in ~30 seconds."
⚠️
Always use --no-cache when building for deployment. Without it, Docker may serve a cached layer that doesn't include your HTML change — the image looks new (new digest) but serves old content. We hit this exact issue in our testing session.
3
Why Pin to Image Digest, Not :latest
⌃

Using :latest in task definitions is a common mistake. ECS can cache the old image and deploy it even after you push a new one to ECR. Pinning to the exact SHA digest guarantees ECS pulls exactly the image you just built:

bash
# BAD — ECS may serve the old cached image
"image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/containers-demo:latest"

# GOOD — ECS must pull exactly this image, no caching possible
"image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/containers-demo@sha256:376ddff..."
💡
In production, use git commit SHA as the image tag: docker build -t myapp:$(git rev-parse --short HEAD) . — every deployment is fully traceable and you can roll back to any previous version instantly.

Scaling In & Out

Operations
1
Manual Scaling
⌃

The simplest scaling — change the desired task count. ECS starts or stops tasks immediately:

bash
# Scale OUT — run 3 tasks (~$0.072/hr total)
aws ecs update-service --cluster containers-demo-cluster \
  --service containers-demo-service --desired-count 3

# Scale IN — back to 1 task
aws ecs update-service --cluster containers-demo-cluster \
  --service containers-demo-service --desired-count 1

# Scale to 0 — stop all tasks, no compute charges
aws ecs update-service --cluster containers-demo-cluster \
  --service containers-demo-service --desired-count 0
✅
Real-world: Duolingo scales non-critical ECS services to 0 overnight and back up in the morning. Saves ~60% on compute costs for batch/background services.
2
Auto Scaling on CPU
⌃

Configure ECS to automatically add tasks when CPU exceeds 70% and remove them when load drops. Real-world use: a daily 9am traffic spike automatically triggers scale-out without any human intervention.

bash
# 1. Register as scalable target (min 1, max 10 tasks)
aws application-autoscaling register-scalable-target \
  --service-namespace ecs \
  --resource-id service/containers-demo-cluster/containers-demo-service \
  --scalable-dimension ecs:service:DesiredCount \
  --min-capacity 1 --max-capacity 10

# 2. Scale out when CPU > 70%, scale in when CPU drops
aws application-autoscaling put-scaling-policy \
  --service-namespace ecs \
  --resource-id service/containers-demo-cluster/containers-demo-service \
  --scalable-dimension ecs:service:DesiredCount \
  --policy-name cpu-target-tracking \
  --policy-type TargetTrackingScaling \
  --target-tracking-scaling-policy-configuration '{
    "TargetValue": 70.0,
    "PredefinedMetricSpecification": {
      "PredefinedMetricType": "ECSServiceAverageCPUUtilization"
    },
    "ScaleOutCooldown": 60,
    "ScaleInCooldown": 300
  }'
⚠️
Always set a max-capacity. Without it, a traffic spike or runaway bug could spin up hundreds of tasks and generate a massive unexpected bill.
3

⌃

Fargate scales linearly — 10 tasks costs exactly 10x one task:

Production Best Practices

Advanced
1
Use an Application Load Balancer (ALB)
⌃

The single most important production upgrade. Direct public IPs change on every task replacement. An ALB gives you a fixed DNS name that never changes and unlocks HTTPS, health checks, and zero-downtime deployments.

FeatureDirect Public IPALB
DNS/AddressChanges on every deployFixed forever
HTTPS/TLSNoYes (free with ACM)
Health checksNoYes — auto-replace unhealthy tasks
Multi-AZNoYes — automatic
Zero-downtime deployNoYes — connection draining
CostFree~$16/month + $0.008/LCU
✅
Used by every production ECS deployment at AWS, Airbnb, Lyft, and Spotify. If your app is public-facing, you need an ALB.
2
Never Use :latest in Production
⌃

Use git commit SHA as image tags. Every deployment is traceable, rollback is instant:

bash
GIT_SHA=$(git rev-parse --short HEAD)
docker build --platform linux/amd64 -t myapp:$GIT_SHA .
docker push $ECR_URI:$GIT_SHA

# Task definition uses specific SHA — fully traceable
"image": "123456789012.dkr.ecr.us-east-1.amazonaws.com/myapp:abc1234"
3
Store Secrets in AWS Secrets Manager
⌃

Never put passwords, API keys, or DB credentials in environment variables as plain text. They appear in CloudWatch logs, ECS console, and CI/CD outputs.

bash
# Store secret — never visible in plain text
aws secretsmanager create-secret \
  --name prod/myapp/db-password \
  --secret-string "mysupersecretpassword"

# Reference in task definition — ECS injects at runtime
"secrets": [{
  "name": "DB_PASSWORD",
  "valueFrom": "arn:aws:secretsmanager:us-east-1:123456789012:secret:prod/myapp/db-password"
}]
⚠️
ECS injects the secret as an environment variable at runtime — it's never stored in the task definition or visible in the console. The container sees it as a normal env var.
4
Add Health Checks
⌃

Without health checks, ECS considers a task healthy as long as the container process is running — even if it's serving 500 errors. With health checks, ECS auto-replaces unhealthy tasks.

json
{
  "healthCheck": {
    "command": ["CMD-SHELL", "curl -f http://localhost/ || exit 1"],
    "interval": 30,
    "timeout": 5,
    "retries": 3,
    "startPeriod": 10
  }
}

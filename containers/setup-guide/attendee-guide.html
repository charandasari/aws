02

Key Concepts Glossary

Beginner
1
Container
⌃

A lightweight, portable package that bundles your app and all its dependencies so it runs the same everywhere. Faster to start than a VM, uses the shared OS kernel rather than a full guest operating system.

💡
Think of it like a shipping container — standardized box, runs on any ship (host machine).
2
Docker Image
⌃

The blueprint/template for a container. Like a class in OOP — you instantiate it to get a running container. Built from a Dockerfile, stored in a registry, pulled when you need to run it.

3
Dockerfile
⌃

A plain text file with step-by-step instructions telling Docker how to build your image. Our demo used just 3 lines:

dockerfile 
FROM nginx:alpine
COPY app/index.html /usr/share/nginx/html/index.html
EXPOSE 80

That's it. Base image, copy your file, open a port.

4
ECR — Amazon Elastic Container Registry
⌃

AWS's private Docker registry. Stores your images so ECS can pull them. Think of it as GitHub for container images — but private and integrated with AWS IAM permissions.

5
ECS — Amazon Elastic Container Service
⌃

AWS's container orchestration service. Decides where and how your containers run, restarts them if they crash, and scales them up or down based on load — all without you having to manage individual servers.

6
Fargate
⌃

A launch type for ECS where AWS manages all the underlying servers. No EC2 to provision, patch, or SSH into. You define CPU and memory; AWS handles the rest.

💡
Fargate is also a launch type for EKS (Kubernetes). Same serverless compute idea, different orchestrator.
7
Task Definition
⌃

A blueprint for your ECS container: image URI, CPU, memory, port mappings, env vars. Think of it as a declarative docker run command stored in AWS. You version task definitions, and ECS uses them to launch containers.

8
ECS Service
⌃

Keeps N copies of your task running at all times. If a container crashes, the service automatically replaces it. Set desired count to 2 and ECS will always maintain 2 healthy running copies — no manual intervention.

Step-by-Step Replication

10 steps · ~30 min
1
Get the Files
⌃

Clone the repo or create the folder structure manually. You need two files: a Dockerfile and the index.html app.

📁 containers/
📄 Dockerfile
📁 app/
📄 index.html
2
Build the Docker Image
⌃

From inside the containers/ directory, build the image and tag it locally as containers-demo.

bash 
docker build -t containers-demo .
Terminal — ~/containers
$ docker build -t containers-demo .
[+] Building 3.2s (7/7) FINISHED
=> [internal] load build definition from Dockerfile
=> [1/2] FROM docker.io/library/nginx:alpine
=> [2/2] COPY app/index.html /usr/share/nginx/html/index.html
=> exporting to image
=> naming to docker.io/library/containers-demo:latest
3
Test Locally
⌃

Run the container locally on port 8080 and verify it works in your browser before pushing to AWS.

bash 
docker run -p 8080:80 containers-demo
# Open http://localhost:8080
# Press Ctrl+C to stop
✅
If it loads in your browser, the container works correctly. Don't push to AWS until this passes.
4
Create ECR Repository
⌃

Create a private repository in ECR to store your image. This is where Docker will push and ECS will pull from.

bash 
aws ecr create-repository \
  --repository-name containers-demo \
  --region us-east-1
💡
Note the repositoryUri in the output — it looks like 123456789012.dkr.ecr.us-east-1.amazonaws.com/containers-demo. You will use it in Steps 6 and 8.
5
Authenticate Docker to ECR
⌃

Docker needs a short-lived token to push to your private ECR registry. This command fetches the token and pipes it directly into Docker login.

bash 
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGION=us-east-1

aws ecr get-login-password --region $REGION \
  | docker login --username AWS --password-stdin \
    $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com
Terminal — ECR Login
Login Succeeded
✅
Token is valid for 12 hours. Re-run this command if you get an authentication error during push.
6
Tag and Push the Image
⌃

Tag the local image with the full ECR URI, then push it to your registry. ECS will pull from this exact URI.

bash 
docker tag containers-demo:latest \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest

docker push \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest
7

⌃

In the AWS Console, navigate to ECS → Clusters → Create Cluster.

FieldValue

InfrastructureAWS Fargate (serverless)

Click Create and wait ~30 seconds for the cluster to provision.

8
Create Task Definition

Navigate to ECS → Task Definitions → Create new task definition. Fill in these fields:

9
Create ECS Service
⌃

Navigate to ECS → Clusters → containers-demo-cluster → Services → Create.

FieldValue

10
Access the App
⌃

Navigate to ECS → Clusters → containers-demo-cluster → Tasks → click the running task → copy the Public IP.

Open http://<public-ip> in your browser.

✅
Your containerized app is live on AWS with no EC2 to manage. The same HTML file, now running in a container on Fargate.

Cost & Cleanup

Always clean up after demos
1
Pricing Breakdown
⌃

Fargate charges for vCPU and memory only while tasks are running. There are no charges when tasks are stopped.

ResourceRate
Fargate vCPU$0.04048 / vCPU / hour
Fargate memory$0.004445 / GB / hour
Combined (0.5 vCPU + 1 GB)~$0.024 / hour while running
ECR image storage$0.10 / GB / month
Fargate when stopped$0.00
⚠️
Always delete your ECS service and cluster after the demo. A running 0.5 vCPU / 1 GB task costs ~$0.024/hour — about $17/month if left running continuously.
2
Cleanup Steps
⌃

Follow these steps in order to avoid any ongoing charges:

  1. Delete the ECS service — ECS → Clusters → containers-demo-cluster → Services → select service → Delete service. Wait for tasks to drain.
  2. Delete the ECS cluster — ECS → Clusters → containers-demo-cluster → Delete cluster.
  3. Delete ECR images — ECR → Repositories → containers-demo → select all images → Delete. This avoids the $0.10/GB/month storage charge.
💡
You can leave the ECR repository itself (empty) at no cost. Only stored image layers incur charges.
06

Next Steps

Where to go from here
1
Try App Runner
⌃

Even simpler than ECS. Point App Runner at your ECR image and it handles load balancing, auto-scaling, and TLS automatically — no clusters, no task definitions, no services to configure.

App Runner is actively maintained and received significant 2025 updates including full VPC support and custom domain configuration. Great next step once you're comfortable with the ECR push workflow.

2
Try EKS
⌃

Deploy the same container using Kubernetes. EKS gives you the full Kubernetes API on AWS-managed control plane — useful when you need advanced scheduling, multi-tenant workloads, or Helm chart ecosystems.

Start with EKS Auto Mode (2025) which manages compute, networking, and storage automatically — dramatically reducing the Kubernetes operational burden. Use eksctl to provision your first cluster in minutes.

3
Add a CI/CD Pipeline
⌃

Automate the entire workflow so every code push deploys itself:

  • Push code to CodeCommit
  • CodeBuild builds the Docker image
  • CodeBuild pushes to ECR
  • CodePipeline triggers ECS service update

Zero manual steps after the pipeline is configured. This is exactly the same CI/CD pattern from the previous session — extended to containers.

4
Add HTTPS
⌃

Put an Application Load Balancer (ALB) with an ACM certificate in front of your ECS service for a real domain and TLS termination.

  • Request a free certificate in ACM
  • Create an ALB pointing at your ECS service target group
  • Add an HTTPS listener (443) using the ACM certificate
  • Point your domain's DNS CNAME at the ALB DNS name

Result: https://yourdomain.com serving your containerized app with zero certificate management overhead.

IAM User & Permissions

Required Before Starting
1
What IAM Permissions You Need
⌃

To run this entire demo on your own AWS account, your IAM user or role needs the following managed policies attached. If you have AdministratorAccess, you already have everything — skip to the next step.

  • AmazonEC2ContainerRegistryFullAccess — create repos, push/pull images, delete images
  • AmazonECS_FullAccess — create clusters, task definitions, and services
  • AmazonVPCFullAccess — view and use VPC, subnets, and security groups
  • IAMFullAccess — create and attach the ECS Task Execution Role (required once during setup)
  • CloudWatchLogsFullAccess — view container logs in CloudWatch
✅
For a personal learning account, attaching AdministratorAccess is the fastest option. In a company account, ask your admin to attach the policies listed above.
2
Create an IAM User (if you don't have one)
⌃

If you are using the AWS root account, create a dedicated IAM user instead. The root account should not be used for day-to-day operations.

  1. Go to IAM → Users → Create user
  2. User name: containers-demo-user (or any name)
  3. Check "Provide user access to the AWS Management Console"
  4. Set a custom password → click Next
  5. Permissions: select "Attach policies directly"
  6. Search and attach each policy: AmazonEC2ContainerRegistryFullAccess, AmazonECS_FullAccess, AmazonVPCFullAccess, IAMFullAccess, CloudWatchLogsFullAccess
  7. Click Next → Create user
⚠️
After creating the user, go to the user → Security credentials → Create access key. Choose "CLI" use case. Download the CSV — you will need this for aws configure.
3
Configure the AWS CLI with your credentials
⌃

Run aws configure in your terminal and enter the credentials from the CSV you downloaded:

bash
aws configure
AWS Access Key ID [None]: AKIAIOSFODNN7EXAMPLE
AWS Secret Access Key [None]: wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY
Default region name [None]: us-east-1
Default output format [None]: json

Verify it works:

bash
aws sts get-caller-identity
Terminal
✓ Returns your Account ID, UserId, and ARN — you are authenticated.
08

ECS Task Execution Role

Required — Create Once Per Account
1
What is the Task Execution Role?
⌃

The ECS Task Execution Role is an IAM role that ECS itself assumes on your behalf when it starts a Fargate task. It gives ECS permission to:

  • Pull your Docker image from ECR (private registry)
  • Write container logs to CloudWatch Logs
  • Pull secrets from Secrets Manager or SSM Parameter Store (if configured)
💡
This role is NOT your user role. It is a role that ECS assumes behind the scenes. Without it, Fargate cannot pull your image from ECR and the task will fail to start.

The standard role name is ecsTaskExecutionRole. AWS creates it automatically in some cases, but you should create it explicitly to be sure.

2
Create the Task Execution Role in IAM Console
⌃

Follow these steps exactly in the AWS Console:

  1. Go to IAM → Roles → Create role
  2. Trusted entity type: AWS service
  3. Service: Elastic Container Service
  4. Use case: Elastic Container Service Task → click Next
  5. Search for and attach: AmazonECSTaskExecutionRolePolicy
  6. Click Next
  7. Role name: ecsTaskExecutionRole
  8. Click Create role
✅
The role name must be ecsTaskExecutionRole — this is the default name ECS looks for. Using a different name means you must manually specify the ARN in every task definition.
3
Check if the Role Already Exists
⌃

If you've used ECS before on this account, the role may already exist. Check via CLI:

bash
aws iam get-role --role-name ecsTaskExecutionRole
Terminal
✓ If it returns a JSON object with the role details, the role exists — skip creation.
⚠ If it returns "NoSuchEntity" error, create the role using the console steps above.
4
Attach the Role to Your Task Definition
⌃

When creating your ECS Task Definition in the console, look for the "Task execution role" field (under the task configuration section). Select ecsTaskExecutionRole from the dropdown.

⚠️
If this field is left blank or set to "None", ECS cannot pull your image from ECR. The task will get stuck in PENDING or go immediately to STOPPED with an authorization error.

The task definition console also has a separate "Task role" field — this is optional and grants permissions to code running inside the container (e.g., accessing S3 from your app). For this demo, leave it blank.

💡
Task execution role = ECS needs to start the task (pull image, write logs). Task role = your app needs to call AWS APIs from inside the container. They are different.

VPC & Networking Setup

Required for Public Access
1
Verify the Default VPC Exists
⌃

Every AWS region comes with a default VPC that has public subnets and an internet gateway already configured. This demo uses the default VPC — no custom networking required.

Verify it exists:

bash
aws ec2 describe-vpcs \
  --filters Name=isDefault,Values=true \
  --query "Vpcs[0].VpcId" \
  --output text
Terminal
✓ Should return something like: vpc-0abc1234def56789
⚠ If it returns "None", create the default VPC: aws ec2 create-default-vpc
2
Verify Public Subnets Exist
⌃

Fargate tasks need to be placed in subnets. For this demo we use public subnets so the task gets a public IP and is reachable from the internet.

bash
aws ec2 describe-subnets \
  --filters Name=default-for-az,Values=true \
  --query "Subnets[*].[SubnetId,AvailabilityZone]" \
  --output table
✅
You should see 2–6 subnets, one per Availability Zone. In the ECS console, select all of them — ECS will pick the best AZ to place your task.
3
Create a Security Group for the Container
⌃

The Fargate task needs a security group that allows inbound HTTP traffic on port 80. Create one in the console:

  1. Go to VPC → Security groups → Create security group
  2. Name: containers-demo-sg
  3. Description: Allow HTTP inbound for ECS Fargate demo
  4. VPC: select the default VPC
  5. Inbound rules → Add rule:
    • Type: Custom TCP
    • Port range: 80
    • Source: Anywhere-IPv4 (0.0.0.0/0)
  6. Outbound rules: leave default (All traffic, 0.0.0.0/0) — ECS needs outbound to pull from ECR
  7. Click Create security group
⚠️
Do not remove the default outbound rule. Fargate needs outbound internet access to pull the image from ECR and send logs to CloudWatch.
4
CloudWatch Logs — View Container Output
⌃

When you create the ECS Task Definition, enable CloudWatch Logs so you can view nginx output and debug issues. In the task definition console:

  1. Under the container configuration, expand Logging
  2. Log driver: awslogs (selected by default)
  3. Log group: /ecs/containers-demo-task (ECS creates this automatically)
  4. Region: us-east-1
  5. Stream prefix: ecs

After the task starts, view logs at CloudWatch → Log groups → /ecs/containers-demo-task.

✅
If a task stops immediately after starting, CloudWatch logs show the exact error — typically a bad image URI, missing file in the container, or a port conflict.

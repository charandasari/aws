01

IAM Permissions

Pre-Demo · Setup
1
Required Managed Policies
⌃

Your AWS account or IAM user needs the following managed policies attached before the session begins. Verify these in the IAM console under your user or role.

  • AmazonEC2ContainerRegistryFullAccess — push/pull/manage ECR repositories and images
  • AmazonECS_FullAccess — create clusters, task definitions, and services
  • AmazonVPCFullAccess — configure security groups and networking for ECS tasks
💡
Admin access also works. In a real environment, scope permissions to least privilege — these managed policies are broader than strictly required for this demo.
2
Verify Permissions Before You Present
⌃

Run a quick smoke test the day before your session to confirm your credentials can reach ECR and ECS. Never discover a permissions issue live in front of an audience.

bash 
# Verify ECR access
aws ecr describe-repositories --region us-east-1

# Verify ECS access
aws ecs list-clusters --region us-east-1

Both commands should return JSON output — not an access-denied error.

Live Demo Commands

During · Command Reference
1
Part 1 — Authenticate, Tag, and Push to ECR
⌃

Copy and run this block in sequence. Always narrate what each command does before pressing enter — do not just run it silently.

bash 
# Capture account ID and region into variables
AWS_ACCOUNT_ID=$(aws sts get-caller-identity --query Account --output text)
REGION=us-east-1

# Authenticate Docker to ECR (token valid 12 hours)
aws ecr get-login-password --region $REGION \
  | docker login --username AWS --password-stdin \
    $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com

# Tag the local image with the full ECR URI
docker tag containers-demo:latest \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest

# Push the image to ECR
docker push \
  $AWS_ACCOUNT_ID.dkr.ecr.$REGION.amazonaws.com/containers-demo:latest
2
Part 2 — Cleanup After the Session
⌃

Run this after the session ends to delete the pushed image and avoid ongoing ECR storage charges. Also stop the ECS service in the console.

bash 
# Delete the pushed image from ECR
aws ecr batch-delete-image \
  --repository-name containers-demo \
  --region us-east-1 \
  --image-ids imageTag=latest
⚠️
Also stop the ECS Service via the console (set desired tasks to 0 or delete the service) to prevent Fargate compute charges from accumulating after the demo ends.

Recovery Tips

If Things Go Wrong
1
Common Problems and Fixes
⌃

If something breaks live, stay calm, narrate what you are investigating, and use this table. The audience learns from watching recovery — it is not a failure.

ProblemFix
ECR auth fails Re-run aws ecr get-login-password. Verify $REGION matches the region where you created the repo. Check that your CLI credentials have not expired.
Task stuck PENDING Check the security group allows inbound TCP 80. Verify the task execution IAM role (ecsTaskExecutionRole) is attached to the task definition.
Task keeps stopping Click the stopped task → Logs tab → read the error message. Common causes: wrong image URI, container crash on startup, or missing environment variables.
Running out of time Skip ECS Service creation. Show the ECR console with the pushed image. Say: "From here, ECS would pull this image, schedule it on Fargate, and expose it via a public IP."
09

Common Beginner Questions

Q&A Prep
1
What's the difference between ECS and EKS?
⌃

ECS is AWS's simpler, fully managed native orchestrator — no Kubernetes knowledge required. Great for teams new to containers.

EKS runs Kubernetes on AWS — more powerful and portable across clouds, but steeper learning curve. In 2025, EKS Auto Mode removes most of the operational overhead: node provisioning, scaling, and upgrades are automated. Current version is 1.35.

Recommended framing: start with ECS; graduate to EKS when you need Kubernetes-native features or multi-cloud portability.

2
Do I need Docker installed on Fargate?
⌃

No. With Fargate, AWS manages the underlying compute entirely. You never SSH into a server, never install Docker on it, and never patch the OS.

You only provide the container image URI. Fargate pulls it from ECR, schedules it on managed infrastructure, and exposes it via the networking config you defined.

3
How much does this cost?
⌃

Fargate charges for the vCPU and memory your task uses while it is running:

  • vCPU: $0.04048 per vCPU per hour
  • Memory: $0.004445 per GB per hour

For this demo (0.5 vCPU / 1 GB): approximately $0.024 per hour. Left running for 24 hours it would cost about $0.58 — but remind attendees to stop the service immediately after the demo.

⚠️
Stop the ECS Service after every demo run. A task left running overnight will accumulate real charges even on a demo account.
4
How does it scale?
⌃

ECS Service Auto Scaling uses CloudWatch alarms to add or remove tasks automatically. You define:

  • Scaling metric: CPU utilization, memory utilization, or custom application metrics
  • Thresholds: e.g., scale out when CPU > 70% for 2 minutes
  • Min / Max task count: the bounds ECS will not cross when scaling

When the alarm triggers, ECS launches new tasks (up to max). When load drops, it terminates tasks (down to min). Fargate provisions and deprovisions compute automatically — no servers to manage.

5
What about HTTPS?
⌃

For production HTTPS, put an Application Load Balancer (ALB) in front of the ECS service. Attach an ACM (AWS Certificate Manager) certificate to the ALB HTTPS listener.

Client (HTTPS)
→
ALB + ACM cert
TLS termination
→
ECS task
HTTP port 80

The ALB handles TLS termination — your container stays simple, serving plain HTTP on port 80. This is the standard production pattern for containerized apps on AWS.

AWS Prerequisites — Roles & Networking
10

ECS Task Execution Role

Create Once Per Account
1
Why This Role Is Required
⌃

The ECS Task Execution Role (ecsTaskExecutionRole) is an IAM role that ECS assumes to perform infrastructure tasks on your behalf when launching a Fargate task:

  • Pull your Docker image from Amazon ECR (your private registry)
  • Write container stdout/stderr to CloudWatch Logs
  • Pull secrets from Secrets Manager or SSM Parameter Store (if configured)
💡
Without this role, Fargate cannot pull the image from ECR. The task will either get stuck in PENDING or fail immediately with an authorization error in the stopped reason. This is the most common first-time setup mistake.

This is different from the Task Role (which grants permissions to code running inside your container). The execution role is infrastructure-level; the task role is application-level.

2
Check if the Role Already Exists
⌃

Run this before the session. If the role exists you can skip creation.

bash
aws iam get-role \
  --role-name ecsTaskExecutionRole \
  --query "Role.Arn" \
  --output text
Terminal
✓ arn:aws:iam::123456789012:role/ecsTaskExecutionRole — role exists, skip to step 4.
⚠ An error occurred (NoSuchEntity) — create the role using steps 3 below.
3
Create the Role via AWS Console
⌃

Navigate: IAM → Roles → Create role

  1. Trusted entity type: AWS service
  2. Service: scroll to Elastic Container Service
  3. Use case: Elastic Container Service Task → click Next
  4. Search permissions: type AmazonECSTaskExecutionRolePolicy → check the box
  5. Click Next
  6. Role name: ecsTaskExecutionRole (exact — ECS looks for this name by default)
  7. Click Create role
✅
Alternatively, create it via CLI in under 30 seconds — use step 4 instead if you prefer the terminal.
4
Create the Role via CLI (Faster)
⌃

Three commands — create the trust policy file, create the role, attach the policy:

bash
# Step 1: Create the trust policy document
cat > /tmp/ecs-trust.json << 'EOF'
{
  "Version": "2012-10-17",
  "Statement": [{
    "Effect": "Allow",
    "Principal": { "Service": "ecs-tasks.amazonaws.com" },
    "Action": "sts:AssumeRole"
  }]
}
EOF

# Step 2: Create the IAM role
aws iam create-role \
  --role-name ecsTaskExecutionRole \
  --assume-role-policy-document file:///tmp/ecs-trust.json

# Step 3: Attach the AWS managed execution policy
aws iam attach-role-policy \
  --role-name ecsTaskExecutionRole \
  --policy-arn arn:aws:iam::aws:policy/service-role/AmazonECSTaskExecutionRolePolicy
✅
Run this as a pre-demo checklist item the day before, not live in front of the audience.
5
Attach the Role in the Task Definition
⌃

When creating the Task Definition in the ECS console, find the "Task execution role" dropdown and select ecsTaskExecutionRole.

⚠️
If the dropdown is empty or the role is missing, it means the role does not exist in this region's account or was created with a different name. Re-run the CLI check in step 2.

During the demo, if a task goes STOPPED immediately — first check: is ecsTaskExecutionRole attached to the task definition? Click the task → Stopped reason → look for CannotPullContainerError or AccessDeniedException.

VPC & Networking Setup

Verify Before the Session
1
Verify the Default VPC and Subnets
⌃

The demo uses the default VPC — every AWS region has one. Run this pre-session to confirm it exists and has public subnets:

bash
# Verify default VPC
aws ec2 describe-vpcs \
  --filters Name=isDefault,Values=true \
  --query "Vpcs[0].{VpcId:VpcId,State:State}" \
  --output table

# Verify public subnets exist
aws ec2 describe-subnets \
  --filters Name=default-for-az,Values=true \
  --query "Subnets[*].[SubnetId,AvailabilityZone,MapPublicIpOnLaunch]" \
  --output table
Terminal — expected output
✓ VPC State: available
✓ 2–6 subnets returned, MapPublicIpOnLaunch = True
⚠ If no subnets: run aws ec2 create-default-subnet --availability-zone us-east-1a
2
Create the Security Group (Pre-Session)
⌃

Create a dedicated security group for the demo container. Do this before the session so you don't create it live (reduces console steps and errors).

bash
# Get the default VPC ID
VPC_ID=$(aws ec2 describe-vpcs \
  --filters Name=isDefault,Values=true \
  --query "Vpcs[0].VpcId" --output text)

# Create the security group
SG_ID=$(aws ec2 create-security-group \
  --group-name containers-demo-sg \
  --description "Allow HTTP port 80 for ECS Fargate demo" \
  --vpc-id $VPC_ID \
  --query GroupId --output text)

# Allow inbound HTTP on port 80
aws ec2 authorize-security-group-ingress \
  --group-id $SG_ID \
  --protocol tcp \
  --port 80 \
  --cidr 0.0.0.0/0

echo "Security Group ID: $SG_ID"
✅
Save the SG_ID value — you'll select this security group when creating the ECS Service during the demo. Do NOT delete the default outbound rule; ECS needs it to pull from ECR.
3
CloudWatch Log Group Setup
⌃

ECS can create the CloudWatch log group automatically when the first task starts, but creating it upfront avoids a common "ResourceNotFoundException" on first launch.

bash
aws logs create-log-group \
  --log-group-name /ecs/containers-demo-task \
  --region us-east-1

In the Task Definition console, enable logging with these settings:

  • Log driver: awslogs
  • awslogs-group: /ecs/containers-demo-task
  • awslogs-region: us-east-1
  • awslogs-stream-prefix: ecs

After the task runs, view logs: CloudWatch → Log groups → /ecs/containers-demo-task. Show this to the audience — it demonstrates that Fargate still gives you full observability without managing servers.

4
Complete Pre-Session Infrastructure Checklist
⌃

Run through all of these the day before the demo:

  • Default VPC exists in us-east-1 (aws ec2 describe-vpcs --filters Name=isDefault,Values=true)
  • At least 2 public subnets exist in the default VPC
  • ecsTaskExecutionRole IAM role exists (aws iam get-role --role-name ecsTaskExecutionRole)
  • Security group containers-demo-sg created with port 80 inbound
  • ECR repository containers-demo created
  • CloudWatch log group /ecs/containers-demo-task created
  • Docker image built and pushed to ECR (pre-push to avoid network issues live)
  • Test task definition created and one task run successfully
⚠️
Do a full dry run the day before — create the cluster, task definition, and service, confirm the app loads via public IP, then delete the service (keep the cluster and task definition). This way the demo infrastructure is validated and you only need to create the service live.
